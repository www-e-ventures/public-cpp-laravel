Laravel-in-C++

The Laravel request lifecycle, compiled.
It's running on this page right now.

A Laravel-shaped web framework in modern C++20 — service container with autowiring, routing + middleware, an Eloquent-style ORM, Blade views, Livewire-style components, auth, sessions, signed tokens, a queue and scheduler. The demo below is not a video and not a mock: it's the actual framework, compiled to a 356 KB WebAssembly module, handling requests inside your browser tab. No server is involved; the request never leaves the page.

Drive it

Try GET /articles (JSON over the ORM), GET /articles.html (a Blade view), GET /articles/1 (route params), or POST /articles with title=Hello&published=1 (validation says no if you drop the title).

Why this exists

It reimplements Laravel's core runtime conventions in C++20 — replicating functionality, not transpiling PHP. Roughly 70% of Laravel is ordinary OOP that ports cleanly; the other 30% is runtime metaprogramming (reflection-driven DI, magic methods, facades) that has to be consciously redesigned. The result keeps Laravel's shape — providers, contracts, middleware pipelines, artisan — with C++'s properties: no source on disk, no eval path, a dispatch table fixed at link time, and a compiled request lifecycle.

  • One lifecycle, many hostsThe same compiled code runs natively, from PHP via FFI, in the browser as WASM, or under a WASM runtime embedded in C++.
  • Zero-dependency coreThe framework library needs the C++ standard library. Period. SQLite and OpenSSL are optional, gated add-ons.
  • Laravel-honest APIsQuery builder, relations, migrations, gates, CSRF, signed tokens (HS256), hashed personal access tokens, DB queue with atomic claims, a cron-style scheduler.
  • Hardened where it countsA throwing handler is a 500, not a dead process; request/WS size limits; CSPRNG sessions with fixation defense; SQL identifier guards; constant-time token compares.
  • Tested, two waysA zero-dependency test harness runs the whole suite in default and Release (-Wall -Wextra -Werror) before any tag.
  • Honest trade-offsWhere C++ can't (or shouldn't) do Laravel magic, the README says so and shows the explicit pattern instead.

In the wild

A 1990s-style bulletin board system — telnet, web, dial-up modem, and LoRa radio on its own epoll + coroutine reactor — vendors the core for its sysop dashboard, JSON API, ORM, identity (hashed tokens, HS256 room tokens, cookie sessions), and daily-turn door games on the scheduler.

A Tandy Color Computer 3/4 emulator compiles to WebAssembly and is served by the framework via FetchContent: the multi-MB .wasm over static serving with Range/ETag, plus versioned binary save-states.

One of them has upgraded across five consecutive releases with zero source changes. And the two apps meet: the BBS streams raw CP437/ANSI over a WebSocket terminal endpoint the emulator's terminal can dial — a genuine CoCo caller on a genuine BBS, in a browser.

Use it

# CMake ≥ 3.21
include(FetchContent)
FetchContent_Declare(lipp
  GIT_REPOSITORY https://github.com/www-e-ventures/public-cpp-laravel.git
  GIT_TAG v0.11.0)
FetchContent_MakeAvailable(lipp)
target_link_libraries(your_app PRIVATE framework)   # + framework_sqlite if you want SQLite

Or clone and run the demo app directly:

git clone https://github.com/www-e-ventures/public-cpp-laravel.git
cd public-cpp-laravel
cmake -S . -B build && cmake --build build && ctest --test-dir build
./build/examples/blog/cpp-artisan serve 8080

The repo's docs/consuming.md covers the three consumption recipes (FetchContent, add_subdirectory vendoring, find_package), and examples/coco-web/ is a complete reference app — static SPA + JSON API + WebSocket terminal.
